# π0.5 (pi05) — an upgraded π0

---

## the one-line version

π0.5 is π0 with a few refinements aimed at **better generalization to genuinely new situations** (new homes, new objects it never saw in training). architecturally it's almost identical to π0 — same PaliGemma + action expert + flow matching — but with a longer language context and a different normalization convention.

read [`pi0.md`](pi0.md) first; this doc only covers the deltas.

---

## what changed from π0

### 1. longer language context (the main config-visible change)

```
π0:    tokenizer_max_length = 48     (short instructions)
π0.5:  tokenizer_max_length = 200    (rich, detailed instructions)
```

π0.5 is built to consume much richer language — not just "pick up the block" but multi-step or detailed instructions, and even high-level subtask plans. the 200-token window reflects that.

### 2. QUANTILE normalization (vs mean-std)

π0 normalizes actions/state with mean-std (subtract mean, divide by std). π0.5 uses **quantile** normalization — it maps values based on their percentile rank in the data distribution rather than assuming a roughly-gaussian shape.

why: robot action distributions are often skewed or have outliers (a gripper that's mostly open or closed, joints that mostly sit in one range). quantile normalization is more robust to that — it doesn't let a few extreme values stretch the whole scale.

> in our wrapper we still apply mean-std ourselves and tell lerobot to treat features as IDENTITY, so this convention difference is mostly internal to how the original model was trained. it matters most if you load official pretrained π0.5 weights.

### 3. the bigger conceptual shift (from the paper)

π0.5's headline contribution is **co-training on heterogeneous data** — it mixes:
- robot demonstrations (like π0)
- high-level semantic subtask prediction (predict "what to do next" in words)
- web data / VQA-style data

the idea is that learning to *describe* tasks in language and predict subtasks helps the model generalize its *actions* to unseen environments. the famous result is π0.5 cleaning kitchens and bedrooms it had never seen in training.

most of that is about the **training data mixture**, not the architecture — so from a wrapper perspective, π0.5 looks almost exactly like π0.

## architecture

identical to π0:

```
PaliGemma 2B (vision-language expert)  +  Gemma 300M (action expert)
                  ↓                              ↓
         shared masked attention
                  ↓
         flow-matching velocity → Euler ODE inference (10 steps)
```

same state/action padding to 32, same flow-matching math, same image [0,1]→[-1,1] preprocessing.

---

## π0 vs π0.5 at a glance

| | π0 | π0.5 |
|---|---|---|
| backbone | PaliGemma 2B | PaliGemma 2B |
| action expert | Gemma 300M | Gemma 300M |
| objective | flow matching | flow matching |
| tokenizer length | 48 | **200** |
| action normalization | mean-std | **quantile** |
| training data | robot demos | **robot demos + subtask prediction + web data** |
| headline | first big flow-matching VLA | **open-world generalization** |

---

## when to use it over π0

- ✓ you have rich/long language instructions or want subtask-level conditioning
- ✓ you care about generalizing to environments not in your training set
- ✓ you're loading official pretrained π0.5 weights
- otherwise π0 and π0.5 behave very similarly for a single fine-tuning task
